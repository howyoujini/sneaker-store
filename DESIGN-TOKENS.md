# Design Tokens — An online store for selling sneakers, editorial and bold, with a hero homepage, product detail pages with photos and size selection, cart, account login, and a simple checkout.

> Generated by the design pipeline (`design_theme`). Code source of truth: `src/theme.css`
> (CSS custom properties, light + dark). To change the look, edit a TOKEN here and in
> `theme.css` — never hardcode a color, size, or radius inside a screen.

## Usage rules
- Style with token classes only: `bg-primary`, `text-muted-foreground`, `border-border`, `rounded-lg` — no raw hex, no arbitrary values.
- Text on a tinted surface uses that surface's `*-foreground` partner (`bg-primary` → `text-primary-foreground`), never a guessed gray.
- Never neutral-gray text on a chromatic background.
- Color dosage ~60/30/10: neutrals carry the UI; `primary` appears only where the user should act.
- Both appearances ship. The preview toggles via `data-theme="dark"` on `<html>` (`?theme=dark` deep link); keep the same mechanism (or a media query) in production.

## Type scale
| Class | Size | Line height | Weight | Role |
|---|---|---|---|---|
| `text-xs` | 11px | 1.5 | 400 | Legal, labels, size chips |
| `text-sm` | 13px | 1.5 | 400 | Captions, secondary metadata |
| `text-base` | 16px | 1.6 | 400 | Body copy, form inputs |
| `text-lg` | 20px | 1.4 | 500 | Section subheadings, cart line items |
| `text-xl` | 26px | 1.25 | 700 | Product name, page titles |
| `text-2xl` | 36px | 1.1 | 800 | Hero callout |
| `text-3xl` | 52px | 1 | 900 | Editorial hero |

- Floors: body (`text-base`) never under 16px; nothing under 11px anywhere.
- These are the ONLY text size classes: the build replaces the Tailwind scale, so `text-4xl` and larger do not exist — hero numbers use `text-2xl`/`text-3xl`.
- In production prefer rem (`size ÷ 16`) so platform text-size settings scale the UI.

## Radius
Base radius `--radius: 6px`. Utilities derive from it (`rounded-md` = radius − 4, `rounded-lg` = radius, `rounded-xl` = radius + 4), so one edit re-rounds the whole product.

## Safe areas (production builds)
- Keep `viewport-fit=cover` on the viewport meta.
- Pad fixed top bars with `env(safe-area-inset-top)` and bottom bars/sheets with `env(safe-area-inset-bottom)`.
- The preview bakes per-device safe-area pixels for its frame; production must use `env()` so real devices drive the insets.
- The preview's `pt-safe` / `pb-safe` classes map 1:1 to `padding-top: env(safe-area-inset-top)` / `padding-bottom: env(safe-area-inset-bottom)` — translate them, don't drop them.
- Keep content out from under the status bar, camera cutout, and home indicator.
- Screens are edge-to-edge: the topmost surface extends under the status bar (its content padded clear), so there is never a strip of bare background behind the clock.
